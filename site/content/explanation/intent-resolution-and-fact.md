## Introduction

When developers discuss game engines, they often begin with state.

There are units, provinces, items, armies, characters, and resources. The database stores their current values, and the engine applies procedures that mutate those values. Combat changes hit points. Movement changes locations. Production changes inventories.

For many games this model is sufficient. For a Play-By-Email (PBEM) game, however, it is often the wrong place to start.

A PBEM turn is fundamentally a process of accepting player intent, resolving conflicts, and producing results. The engine is not primarily concerned with what the world looks like. It is concerned with how the world changes.

The organizing principle of a PBEM engine should therefore be an event stream.

More precisely, it should be a system that transforms intent into fact.

## Orders Are Intent

Consider two players.

Anne controls Biff.

Joe controls Skar.

Anne submits:

```text
MOVE 3,4
ATTACK ANYONE
```

Joe submits:

```text
MOVE 3,4
DEFEND
```

The order parser validates the syntax and performs shallow semantic checks.

- Does the unit exist?
- Can it move?
- Can it attack?
- Is the destination valid?

The parser does not resolve combat.

The parser does not move units.

The parser does not modify game state.

Instead, it emits intent records:

```text
MoveIntent(Biff, 3,4)
AttackIntent(Biff, TargetOfOpportunity)

MoveIntent(Skar, 3,4)
DefendIntent(Skar)
```

These records represent accepted player intent.

Nothing has happened yet.

## The Engine Resolves Intent

The engine operates in phases.

Each phase consumes certain intents, resolves them against a view of the world, and emits facts.

The movement phase might consume:

```text
MoveIntent(Biff, 3,4)
MoveIntent(Skar, 3,4)
```

and emit:

```text
BiffArrived(3,4)
SkarArrived(3,4)
```

The defense phase might consume:

```text
DefendIntent(Skar)
```

and emit:

```text
SkarDefending(+18)
```

The attack phase consumes:

```text
AttackIntent(Biff, TargetOfOpportunity)
```

and emits:

```text
CombatOccurred(...)
BiffWounded(...)
SkarGainedExperience(...)
```

The distinction is important.

Intent describes what a player wants.

Fact describes what actually happened.

The engine exists to transform one into the other.

## Every Phase Is a Small Engine

Traditional game engines are often implemented as large procedural systems.

A PBEM engine benefits from a different approach.

Each turn phase can be treated as a small engine.

Its responsibilities are simple:

1. Read relevant records.
2. Resolve them.
3. Emit results.

Movement becomes one engine.

Combat becomes another.

Production becomes another.

Magic becomes another.

Each phase is responsible for a narrow set of rules.

This decomposition has two advantages.

First, it makes the system easier to reason about.

Second, it makes the system easier to test.

## Derived Intent

Not every resolution produces only facts.

Sometimes a resolution creates additional work.

Suppose combat determines that Biff must retreat.

The combat phase may emit:

```text
RetreatRequired(Biff)
```

This is not a player order.

It is not a completed fact.

It is intent generated by the simulation itself.

A later phase consumes that intent and emits:

```text
BiffRetreated(3,4 -> 3,5)
```

The engine therefore operates as a pipeline:

```text
Intent
    ↓
Resolution
    ↓
Fact
    ↓
Derived Intent
    ↓
Resolution
    ↓
Fact
```

The process continues until no unresolved intents remain.

## The Event Stream Is the Narrative

One of the most valuable properties of this model is that it leaves behind a story.

Traditional state-oriented systems often answer the question:

> What does the world look like now?

An event-oriented system also answers:

> Why does the world look this way?

A report writer can reconstruct player-visible results.

A developer can reconstruct a bug.

A game master can reconstruct a disputed outcome.

The stream becomes a narrative of the turn.

That narrative is useful for debugging, reporting, testing, and auditing.

## State Still Matters

This does not mean state disappears.

The engine still requires state in order to resolve events.

Combat illustrates the point.

When Biff fights Skar, combat should operate against a stable set of inputs:

```text
Biff attributes
Skar attributes
Terrain
Modifiers
Weather
```

Those values should not change halfway through resolution.

Combat therefore consumes a snapshot of relevant state and produces results.

The event stream explains how state changes.

The state provides the context required to resolve those changes.

The two are complementary.

## Restartability

An interesting consequence of this model is that the engine itself never needs to understand restarting.

The engine's job is simple:

```text
while unresolved work exists:
    resolve next item
```

Administrative actions such as rewinding a turn become operations on the stream rather than operations on the engine.

The game master may choose to discard derived results and replay resolution.

The engine simply resumes processing work.

From its perspective, nothing special happened.

## Testing

This architecture is particularly attractive for automated testing.

Each phase consumes known inputs and emits known outputs.

A test can describe behavior in domain language:

```gherkin
Given Biff is in province 3,3
And Skar is in province 3,4

When Biff submits a move order to 3,4
And the movement phase executes

Then Biff arrives in province 3,4
```

The test focuses on intent and outcome rather than implementation details.

Because the event stream captures both, verification becomes straightforward.

## Conclusion

A PBEM engine should not be organized around state mutations.

It should be organized around the transformation of intent into fact.

Player orders become intent.

Turn phases resolve intent.

Resolution produces facts and occasionally new intent.

State provides context.

The event stream provides explanation.

Viewed this way, the engine is not a collection of procedures that modify a database. It is a system that continuously consumes intent, resolves conflicts, and records what happened.

That record becomes the history of the turn, the basis of testing, the source of reports, and the foundation of the game's rules.

The event stream is not merely a logging mechanism.

It is the organizing principle of the engine.
